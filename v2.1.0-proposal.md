# v2.1.0 Proposal — Deep Analysis

**Date:** 2026-07-18
**Target version:** 2.1.0
**Status:** Draft proposal for prioritization

---

## 1. Executive Summary

PgSchemaExporter v2.0.0 delivered the architecture refactor (`ICommand` / `CommandDispatcher`), source-generated JSON, atomic `MigrationHistory`, SQL escaping, benchmark infrastructure, and a unified `SqlTokenizer`/`SqlStatementCache` foundation. v2.1.0 should focus on **three pillars**:

1. **Completing the SQL parser unification** so every parser (`TableDefinitionParser`, `ConstraintDefinitionParser`, `SqlDropBuilder`, `ErModelBuilder`) consumes a single, token-level API and is backed by dense unit tests.
2. **Hardening CI and automation** with benchmark baselines, regression detection, and GitHub Actions-native reporting.
3. **Production safety and operability** for migrations: a migration journal, idempotent `apply`, and thread-safe progress reporting.

This document lists candidate features, groups them by theme, proposes a priority order, and flags effort, risk, and value for each item.

---

## 2. What v2.0.0 Already Covered

From `README.md` and `release_notes_2.0.0.md`:

- CLI architecture rebuilt (`ICommand` handlers, `CommandDispatcher`).
- `EffectiveSchemas` and non-mutating `ExportOptions` validation.
- Atomic `MigrationHistory.AppendAsync`.
- Source-generated JSON (`PgSchemaExporterJsonContext`) for `MigrationPlan`, `MigrationHistory`, `SchemaDiffReport`, `SchemaFingerprint`.
- SQL identifier / timeout-value escaping in `MigrationScript` and `TableMigrationBuilder`.
- `CommandTimeout` and `MigrationJournal` in `MigrationApplier`.
- Streaming reads in `DumpSplitter` and `SchemaDiffer` concurrency (`ConcurrentBag`).
- `SqlTokenizer` / `SqlStatementCache` foundation.
- `PgSchemaExporter.Benchmarks` suite and `.github/workflows/benchmarks.yml`.
- `PgSchemaExporter.slnx` and unified `2.0.0` versioning.

The remaining gaps are mostly **incremental correctness, test coverage, and operational ergonomics** rather than big architectural rewrites.

## 3. Strategic Focus for v2.1.0

| # | Focus | Why it matters |
|---|---|---|
| 1 | **Token-level parser unification** | `SqlTokenizer` already splits statements and finds parens; individual parsers still do their own `IndexOfWord`/`FindKeyword` scanning. Consolidating them removes the last duplicated quote/comment logic and is the only way to safely support nested SQL (e.g. `CREATE POLICY`, `RULE`, table inheritance). |
| 2 | **CI benchmark regression detection** | Benchmarks run on demand and upload artifacts, but PRs are not automatically compared to a baseline. This is the best defense against silent perf regressions in `DeploymentPlanBuilder`, `PostgresMetadataProvider`, and `MigrationGenerator`. |
| 3 | **Migration safety & observability** | `MigrationApplier` can partially apply a plan when `CREATE INDEX CONCURRENTLY` fails. A journal and idempotent re-apply are needed for production use. |
| 4 | **Metadata & export robustness** | `PolicyDefFunctionExistsCache` is static and never invalidated; `PostgresMetadataProvider` still uses `information_schema` in `GetCommentsAsync`/`GetGrantsAsync`; `ExportConfigLoader` uses reflection (`DefaultJsonTypeInfoResolver`), breaking AOT/trimming. |
| 5 | **CLI / packaging polish** | Make `pgschema-export` installable as a global .NET tool, improve `init` scaffolding, and emit GitHub Actions summaries from `drift` and `diff`. |

## 6. Implementation Phases

A realistic v2.1.0 cycle can be split into two milestones.

### Milestone 1 — Parser & CI Foundation (weeks 1–3)

1. Add token stream API to `SqlTokenizer` (P.1).
2. Port `TableDefinitionParser` and `ConstraintDefinitionParser` (P.2, P.3).
3. Add parser unit tests (P.5).
4. Set up benchmark baseline artifact and PR comparison (C.1).
5. Fix thread-safe `ConsoleProgressReporter` (D.3) and `PolicyDefFunctionExistsCache` (E.1).

**Deliverable:** `main` has a cleaner parser core, faster CI signal, and regression protection.

### Milestone 2 — Production & Packaging (weeks 4–6)

1. Implement migration journal and idempotent `apply` (M.1, M.2).
2. Profile and optimize `GetCommentsAsync`/`GetGrantsAsync` (E.2).
3. Move `ExportConfigLoader` to source-gen JSON (E.3).
4. Package CLI as a .NET global tool (D.1).
5. Add GitHub Actions summaries for `diff`/`drift` (C.2).
6. Optional: JSON manifest or `init` scaffolding.

**Deliverable:** v2.1.0 is releasable, safer for production migrations, and easier to install.

## 7. Risks & Dependencies

| Risk | Mitigation |
|---|---|
| **Token-level parser rewrite could introduce regressions** | Keep the old implementation behind a feature flag for one release, or run both and compare outputs on large fixtures. |
| **Migration journal changes the `apply` contract** | Make journal opt-in first (`--journal`), then default in a later release. |
| **Benchmark baseline storage grows** | Store only summary metrics (`Mean`, `Allocated`, `Gen0/1/2`) and the raw artifact only for the last `main` run. |
| **AOT/trimming for config loader may break plugins** | Add a small IL trimming test before enabling `PublishTrimmed`. |
| **Global tool packaging competes with custom build scripts** | Keep `dotnet publish` self-contained path unchanged; tool package is an additional output. |

---

## 8. What to Defer to v2.2.0 or Later

Not everything fits in one release. Consider moving these out of v2.1.0:

- **Streaming archive output** (O.2): requires refactoring `SchemaFileWriter` into a stream abstraction; lower immediate value than parser and CI.
- **Partitioned tables / FDW** (PG.2, PG.3): deep PostgreSQL features that need dedicated fixtures and testing.
- **Schema glob patterns** (D.4): useful but can be added without breaking changes.
- **Rollback verification** (M.3): depends on a stable journal implementation; better after M.1/M.2 have baked.
- **Release workflow automation** (C.4): can remain manual until release cadence increases.

## 9. Success Metrics

At the end of v2.1.0, we should be able to measure:

- **Parser correctness:** unit tests cover `SqlTokenizer`, `TableDefinitionParser`, `ConstraintDefinitionParser`, and `MigrationGenerator` without Testcontainers; code coverage for `Scripting` namespace > 80 %.
- **CI regression:** `benchmarks.yml` fails the PR if any benchmark regresses > 10 % mean time or > 20 % allocated memory.
- **Migration safety:** `MigrationApplier` leaves a journal row for every executed statement; `--resume` passes an integration test that replays a failed migration.
- **Distribution:** `dotnet tool install -g pgschema-export` works from a NuGet feed.
- **Performance:** `PostgresMetadataProviderBenchmark.LoadMetadata` does not regress; `GetCommentsAsync`/`GetGrantsAsync` profiling notes are documented in `doc/`.

---

## 10. Conclusion

v2.1.0 should be a **correctness + operability** release, not another architecture overhaul. The most important deliverables are:

1. Finish the token-level parser unification started in v2.0.0.
2. Guard performance with benchmark baselines in CI.
3. Add a migration journal so `apply` is safe in production.
4. Polish packaging (global .NET tool) and config serialization.

This scope is large enough to be meaningful, small enough to ship in 6 weeks, and directly addresses the remaining risks from `ASSESSMENT_REPORT.md` and the `Performance_Refactoring_Report.md` "Future Directions" section.
